# OctoAcme Project Management Overview

OctoAcme runs projects with an iterative, customer-first lifecycle that is intentionally lightweight but explicit about ownership and measurable outcomes. Work moves through five phases: **Initiation → Planning → Execution/Tracking → Release/Deployment → Close/Retro**. In **Initiation**, teams validate the business need and align stakeholders using a **Project One-pager** that captures the problem, goal, and success metrics, alongside a stakeholder list, communication plan, high-level timeline, initial risk list, and resource needs. A decision gate advances the project only when success metrics are clear, stakeholders agree on priority, and team availability is confirmed. During **Planning**, the team holds a kickoff, builds a prioritized backlog with acceptance criteria, estimates scope using T-shirt sizing or story points, documents a **Definition of Done**, and maps milestones and releases. Risks and cross-team dependencies are tracked in a **Risk Register** (recording impact, likelihood, owner, mitigation, and status) and reviewed regularly throughout the project.

Roles are explicitly defined to reduce ambiguity and improve throughput. The **Project Manager (PM)** coordinates delivery, timelines, risks, and communications. The **Product Manager (PdM)** defines outcomes, prioritizes the backlog, and measures success. **Developers** implement and test changes while collaborating on design and estimates. **QA/Testing** validates acceptance criteria and release readiness. **Stakeholders** provide input and approvals. Day-to-day execution uses a project board workflow—**Backlog → Ready → In Progress → In Review → QA → Done**—supported by a team rhythm of daily standups, a weekly delivery sync, and a demo/review at the end of each sprint or milestone. Progress is tracked through delivery metrics (velocity, burndown) and the product success metrics defined during initiation, with dashboards monitoring errors, latency, and usage.

Communication and quality practices are embedded in the workflow. Stakeholder updates follow a regular cadence using a standard weekly status template covering progress, next steps, risks/blockers, and asks/decisions, with a **single source of truth** (the project README or release doc) for current status. Blockers follow a clear **escalation path**: team triage → PM escalation → Product Lead and dependent teams → sponsor-level escalation for major business impact, with a separate route to the on-call security team for security incidents. Quality is enforced through the PR and CI process: PRs target **≤ ~400 lines**, include an issue link and acceptance criteria, pass **automated tests, linting, and security scanning in CI**, and require **at least one approval** before merge.

Releases follow standardized pre-release checks—acceptance criteria met, CI/security scans passing, release notes written, rollback plan documented, and smoke tests executed—before progressing through a staged deployment checklist that includes staging validation, post-deploy verification, and stakeholder announcements. Testing expectations cover **unit tests for new logic**, **integration tests where applicable**, and **end-to-end smoke tests for critical flows**, with **manual QA for feature acceptance** when needed. After each release, teams run a blameless retrospective (45–75 minutes, with an optional anonymous input board) and prioritize **2–3 action items** with assigned owners tracked in the backlog or issues. Progress on those items is reviewed in the weekly PM sync, completing the continuous improvement loop that anchors OctoAcme's delivery culture in psychological safety and data-informed decisions.
